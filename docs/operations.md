# 運用方針 / 配布モデル

バックエンドサーバの運用と、modを公開した際にクライアント（mod）が「対応サーバを見つける」方法に関する設計判断。

---

## 配布モデルの選択肢

### A. 公開単一サーバ（mod作者がホスト）
mod に `STS_STATS_BACKEND_URL=https://<author-domain>` をデフォルト埋め込み。

- ✅ 仲間側ゼロ設定
- ✅ 共有URLのドメインが固定で覚えやすい
- ❌ 作者が運用責任を持つ（落ちたら全員影響）
- ❌ 費用負担（無料tier内なら問題無し）
- ❌ 悪用リスク（POST `/sessions` 無認証）

### B. セルフホスト（各ユーザー）
- ✅ 作者は運用から解放
- ❌ Web運用知識ない人には敷居高い → mod採用率低下
- ❌ ホストごとに共有URLドメインがバラバラで統一感なし

### C. ハイブリッド（採用方針）
**デフォルトは作者の公開サーバ、`.env`/設定ファイルで上書き可能**。

- 一般ユーザー → A の体験
- 開発者・上級者 → 自前サーバに向けられる
- バックエンド消滅時 → JSONL ローカルログだけは残る

「**対応サーバの存在を知る方法**」は、このモデルだと **mod に固定URLを焼き付ける** 形で解決。サービスディスカバリー（DNS TXT 等）はオーバースペック。

---

## 設計時に守ること（公開判断を遅らせるための仕込み）

Phase 2 段階ではローカルサーバ前提で進めて良いが、**以下の3つは設計時点で満たしておく**。これにより後で公開モードへ切り替える際の手戻りがほぼゼロになる。

1. **バックエンドURLは config で上書き可能にする**（再ビルド不要）
   - `.env` だけだと再ビルドが必要なので、ゲームディレクトリ側の `mods/StsStats/config.json` も読むようにする
2. **バックエンド消滅でもゲームを止めない**（既に実装済み: HTTP失敗時はログのみで継続）
3. **共有URLパスは `/s/<uuid>` 固定**（ドメイン切替に強くする）

## 公開時のチェックリスト（将来用）

- [ ] **独自ドメイン**で本番サーバを立てる（`sts2stats.fly.dev` 直ではなく `sts2stats.example.com`）。Fly/Render 等が消えても DNS 切替で延命可能
- [ ] mod のデフォルト `STS_STATS_BACKEND_URL` を本番URLに
- [ ] mod 起動ログに「セルフホストしたい人は config.json で上書き可」を明示
- [ ] README に Docker一発起動の**セルフホスト手順**を記載
- [ ] バックエンド側で **rate limit**（`chi` + `httprate`、IP単位 `POST /sessions` 10/分等）
- [ ] **データ保持期限**（30日経過セッション自動削除）でストレージ増殖を防ぐ
- [ ] 個人特定情報（NetId 以外）を扱わないことの明文化

---

## バックアップとフェイルセーフ

- mod は HTTP 送信と**並行して** `sts_stats.jsonl` ローカルログを残す（実装済み）。バックエンド消滅・通信不可でも個人記録は失われない
- 共有URL機能だけがオプション扱いに降格できる設計
